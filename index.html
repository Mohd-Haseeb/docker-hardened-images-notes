01 — The core idea
What does "hardened" actually mean?

The word gets used loosely. Here's the precise definition and the mental model that makes everything else click.

The analogy

Imagine two offices. The first has desks, computers, a kitchen, a supply closet full of tools, and a loading dock with a van. If someone breaks in, they have everything needed to steal, communicate, and escape. The second office has only a single desk and one computer — nothing else. Both offices do the same job. But a break-in to the second yields almost nothing. That's the difference between a standard image and a hardened image.

⬡
Minimal footprint
Only the runtime your app needs. No shell (bash, sh), no package manager (apt, pip), no build tools (gcc). Every removed component is one fewer attack surface.
🔒
Zero CVEs at ship time
Docker scans every image before publishing. All known critical and high CVEs are patched. Released only when the scan is clean — unlike standard distro images which ship with whatever CVE state Debian happens to be in.
✍️
Signed with provenance
Every image is signed via Sigstore/Cosign and ships with an SBOM — a cryptographic bill of materials proving exactly what's inside and how it was built. Not just what the content is, but that it hasn't been tampered with.
👤
Non-root by default
Runs as UID 65532 (nobody). Standard images run as root inside the container. If a container escapes its namespace, running as root is catastrophic. An unprivileged UID dramatically limits the blast radius.
📁
Read-only filesystem
Designed to run with --read-only flag. Image layers cannot be modified at runtime. Malware that writes executables to disk is stopped cold. Your app gets a writable /tmp tmpfs mount only.
🔁
Continuously rebuilt
When a new CVE drops against any package in the image, Docker's automation triggers a rebuild within hours — not days or weeks typical of distro security channels. Pull the new tag and you're clean.

The key insight: Hardening doesn't prevent the initial breach — your app can still have an RCE bug. What it does is contain the blast radius. The attacker gets in but has almost nothing to work with. What would be a full compromise becomes a dead end.
03 — Technical learning notes
Key concepts consolidated

Everything from our discussion, organised for reference and revision.

NOTE 01 — ARCHITECTURE
The five security properties
  • Distroless base: Built on scratch or Google's distroless base. No Debian/Ubuntu package ecosystem. No shell binary anywhere in the image.
  • CVE-free at release: Docker scans against NVD, OSV, GitHub Advisory before publishing. Only released when critical/high CVE count is zero.
  • Signed + SBOM: Cosign signature + embedded SBOM. You can cryptographically verify the image wasn't tampered with between Docker's build infra and your runtime.
  • Non-root UID: Runs as UID 65532 (nobody) by default. No capability elevation. Drops NET_ADMIN, SYS_PTRACE, CAP_SETUID.
  • Read-only filesystem compatible: Designed for --read-only. App gets a writable /tmp tmpfs only. Defeats runtime malware that writes executables to disk.
NOTE 02 — IMAGE VARIANTS
Three tiers, one promotion path

DHI ships in three variants. The intended workflow is: develop locally with dev, debug issues in staging with debug, ship only minimal to production.

VariantContentsUse caseAttack surface
minimalRuntime only. No shell, no pkg manager.ProductionSmallest
debugMinimal + busybox shell + basic net toolsStaging / triageMedium
devDebug + package manager + build toolsLocal developmentLargest

The key rule: never ship the debug or dev variant to production. busybox alone provides 300+ Unix utilities in a single binary — one binary undoes the most critical protection.

  • SLSA Level 3: Supply chain Levels for Software Artifacts. DHI targets L3 — the build process itself is trustworthy, audited, and non-forgeable.
  • Reproducible builds: Same inputs → byte-for-byte identical output. You can verify the published image was produced from the declared source.
  • Hermetic builds: Build environment has no internet access. Every dependency fetched from a controlled mirror. Prevents dependency confusion attacks.
  • Cosign + Sigstore: Keyless signing tied to OIDC identity. No long-lived private keys to steal or rotate.
NOTE 04 — BLAST RADIUS
Why "no shell" is the single biggest win

Almost every known container attack chain requires a shell for at least one step. Without it, attackers are forced to work entirely within your application's language runtime, which dramatically narrows their options.

With shell present
  • Run arbitrary OS commands
  • Invoke curl/wget for C2 comms
  • Write and execute scripts
  • Modify /etc/passwd, cron
  • Pivot using SSH, nc, socat
Without shell
  • Limited to Python/Node runtime APIs
  • No raw OS command execution
  • All activity visible in app logs
  • Cannot write to filesystem
  • Network calls hit egress policies
NOTE 05 — AZURE INTEGRATION
How DHI fits your Azure upskilling path
AKS — Admission control
Enforce image signature verification at admission time via Azure Policy + OPA Gatekeeper. DHI's Cosign signatures are verifiable before any pod is scheduled.
ACR — Image mirroring
Mirror DHI into ACR using az acr import. ACR's Defender for Containers shows near-zero findings on DHI workloads, directly improving your Defender for Cloud secure score.
Defender for Containers
DHI's SBOM gives Defender richer context for vulnerability assessment. Instead of scanning an opaque layer, Defender matches known packages against its threat database precisely.
Azure DevOps / Pipelines
Use docker scout as a quality gate. Compare your image against DHI equivalents and fail builds that exceed a CVE threshold — enforcing security at the CI layer.
WAF Security Pillar
DHI satisfies WAF requirements SE:01 (minimise attack surface), SE:05 (least privilege), SE:11 (image hygiene). Maps cleanly to AZ-305 security architecture questions.
VNet + Private Endpoints
DHI's inability to use curl/wget complements VNet egress controls — defence in depth. Even if egress rules are misconfigured, the container has no tools to exploit it.
NOTE 06 — TRADEOFFS
What you give up and how to handle it
  • Debugging is harder: No kubectl exec + shell in prod. Use kubectl debug to attach an ephemeral debug sidecar in staging. Never leave debug containers running.
  • Coverage gaps: Not every base image has a hardened counterpart yet. Build your own using distroless base + Cosign + SBOM generation via Syft for niche runtimes.
  • App assumes writable filesystem: Frameworks that unpack to disk at startup need refactoring. Move write operations to an explicitly mounted volume or tmpfs.
  • Startup processes: Apps that run pip install at startup (common anti-pattern) will break. Dependencies must be baked into the image at build time — which is correct practice anyway.
  • Licensing: DHI requires a Docker subscription for commercial use beyond a certain scale — similar to Docker Desktop's licensing model.
04 — Quick reference
Commands and patterns

The commands and Dockerfile patterns you'll reach for most often.

PULLING AND VERIFYING A DHI
# Pull the minimal (production) variant
docker pull docker.io/docker/python-hardened:3.12-minimal

# Verify the Cosign signature
cosign verify \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  docker.io/docker/python-hardened:3.12-minimal

# Inspect the SBOM
docker scout sbom docker/python-hardened:3.12-minimal
DOCKERFILE — MULTI-STAGE BUILD
# Stage 1: build (full tools available)
FROM python:3.12 AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

# Stage 2: runtime (hardened — nothing extra)
FROM docker/python-hardened:3.12-minimal
WORKDIR /app
COPY --from=builder /app /app
USER 65532
ENTRYPOINT ["python", "app.py"]
RUNNING WITH SECURITY FLAGS
docker run \
  --read-only \        # read-only root fs
  --tmpfs /tmp \      # writable scratch only
  --user 65532:65532 \ # non-root user
  --cap-drop ALL \    # drop all capabilities
  --security-opt no-new-privileges \
  myapp:latest
AZURE — ACR IMPORT + CVE GATE
# Mirror DHI into your ACR
az acr import \
  --name myregistry \
  --source docker.io/docker/python-hardened:3.12-minimal \
  --image python-hardened:3.12-minimal

# Fail build if any CRITICAL/HIGH CVEs found
docker scout cves $(IMAGE):$(TAG) \
  --exit-code \
  --only-severity critical,high
AKS — ENFORCE IMAGE SIGNING POLICY
# Enable Ratify for signature verification
az k8s-extension create \
  --cluster-name myaks \
  --extension-type microsoft.flux \
  --name ratify

# Apply "only signed images" Azure Policy
az policy assignment create \
  --policy "Kubernetes clusters should
            only use allowed images"
DEBUG A HARDENED CONTAINER
# Attach ephemeral debug container to a running pod
kubectl debug -it pod/mypod \
  --image=docker/python-hardened:3.12-debug \
  --target=mycontainer

# Attaches a debug sidecar without modifying
# the prod container. Discard immediately when done.
# NEVER use this in prod for longer than necessary.
